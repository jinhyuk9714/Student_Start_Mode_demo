## 1. State Machine: 체크리스트 항목 상태 전이

### 상태 정의

| 상태 | 설명 | UI 표현 |
|------|------|---------|
| `LOCKED` | 선행조건 미충족. 시작 불가 | 잠김 아이콘 + "먼저 해야 할 것" 표시 |
| `AVAILABLE` | 모든 선행조건 충족. 지금 시작 가능 | 활성 카드 + "시작하기" 버튼 |
| `IN_PROGRESS` | 사용자가 진행 중 | 진행 중 표시 + "완료" 버튼 |
| `COMPLETED` | 완료됨 | 체크 표시 + 접힌 상태 |
| `REMOVED` | 조건 변경으로 더 이상 해당 없음 (비자 변경 등) | 취소선 + 회색 처리, 접힌 상태 |

### 전이 다이어그램

```
                    ┌──────────────────────────────────────────────┐
                    │                                              │
                    ▼                                              │
               ┌─────────┐    prerequisites met    ┌───────────┐  │
  (생성시) ──→ │ LOCKED   │ ─────────────────────→ │ AVAILABLE │  │
               │ (잠김)   │                         │ (가능)    │  │
               └────┬────┘                         └─────┬─────┘  │
                    │▲                                    │        │
                    ││ prerequisites                      │ user   │
                    ││ invalidated                        │ starts │
                    ││                                    ▼        │
                    ││                               ┌───────────┐  │
                    ││                               │IN_PROGRESS│  │
                    ││                               │ (진행중)  │  │
                    ││                               └─────┬─────┘  │
                    ││                                     │        │
                    ││                                     │ 확인   │
                    ││                                     │ (버튼/ │
                    ││                                     │  AI)   │
                    ││                                     ▼        │
                    ││                               ┌───────────┐  │
                    ││                               │ COMPLETED │  │
                    ││                               │ (완료)    │──┘
                    ││                               └─────┬─────┘
                    ││        status change               │ triggers
                    │└────────(비자 변경 등)───────────────┘ cascade
                    │                                       unlock
                    │
                    │  조건 변경으로 해당 없음
                    ▼
               ┌─────────┐
               │ REMOVED  │  ※ COMPLETED도 조건 변경 시 REMOVED 가능
               │ (제거됨) │  ※ 새 조건이 다시 충족되면 LOCKED로 복귀
               └─────────┘
```

### 전이 규칙

| 전이 | 트리거 | API |
|------|--------|-----|
| LOCKED → AVAILABLE | 모든 prerequisite 충족 (Rule Engine cascade) | 자동 |
| AVAILABLE → IN_PROGRESS | 사용자 "시작" 클릭 | `POST /api/progress {action: "start"}` |
| IN_PROGRESS → COMPLETED | 사용자 확인 (아래 완료 메커니즘 참조) | `POST /api/progress {action: "complete"}` |
| COMPLETED → cascade unlock | 이 항목이 다른 항목의 prerequisite | Rule Engine 자동 |
| AVAILABLE → LOCKED | prerequisite 무효화 (드문 케이스) | Rule Engine 재평가 |
| * → REMOVED | required_condition이 더 이상 충족되지 않음 (비자 변경 등) | Rule Engine 재평가 |
| REMOVED → LOCKED | 조건이 다시 충족됨 | Rule Engine 재평가 |

### 금지된 전이 (차단)

- ✗ LOCKED → COMPLETED (건너뛰기 불가)
- ✗ LOCKED → IN_PROGRESS (선행조건 미충족 상태에서 시작 불가)
- ✗ COMPLETED → AVAILABLE (완료 → 가능으로 되돌리기 불가, 단 REMOVED는 가능)

### 완료 메커니즘: AI 감지 + 확인 카드

체크리스트 항목의 완료는 **항상 사용자의 명시적 확인**이 필요하다. AI가 자동으로 체크하지 않는다.

**완료 경로 2가지:**

```
경로 A: 체크리스트 UI에서 직접 완료
────────────────────────────────────
사용자 → 체크리스트 화면 → 항목의 "완료" 버튼 클릭
→ POST /api/progress/:sid {action: "complete", code: "OPEN_BANK_ACCOUNT"}
→ cascade unlock

경로 B: AI 채팅에서 감지 → 인라인 확인 카드 (권장)
────────────────────────────────────
사용자: "계좌 만들었어요!"
→ LLM이 완료 의도 감지
→ pending_completion 응답 (아직 실제 완료 처리 안 함)
→ 프론트엔드: 인라인 확인 카드 렌더링

┌─────────────────────────────────────────┐
│ 🤖 축하해요! 계좌 개설 완료로             │
│    체크할까요?                            │
│                                         │
│  ☑ 하나은행 계좌 개설                     │
│                                         │
│  [✓ 완료 확인]    [아직 아니에요]           │
└─────────────────────────────────────────┘

→ "완료 확인" 탭 → POST /api/progress/:sid {action: "complete", code: "..."}
→ cascade unlock
→ 채팅에 unlock 결과 표시: "이제 2가지가 가능해졌어요!"
```

**왜 AI 자동체크가 아닌가?**
- AI가 "계좌 만들었어요"를 오판할 수 있음 (예: "계좌 만들었어요?"라는 질문)
- 사용자가 잘못 말한 경우에도 한 번 더 확인 기회
- 확인 카드 UX로 유저 통제감 유지 + 환각 리스크 원천 차단
- "아직 아니에요" 선택 시 아무 상태 변경 없이 대화 계속

## 2. Rule Engine: 5단계 실행 로직

### 입력

```json
{
  "user_profile": {
    "university_id": 1,        // 서울대
    "visa_type_id": 1,         // D-2
    "nationality": "CN",       // 중국
    "arc_applied_date": "2026-04-02"  // ARC 신청일 (선제 프롬프트용)
  },
  "user_conditions": {         // user_conditions 테이블에서 조회
    "HAS_ARC": false,
    "HAS_PHONE": true,         // 선불폰
    "HAS_PHONE_POSTPAID": false,
    "HAS_ACCOUNT": false,
    "HAS_CARD": false,
    "HAS_SCHOLARSHIP": false   // 장학금 수령 예정 여부
  }
}
```

### 스키마 추가 (페르소나 검증 + 완료/동적추가 설계로 발견)

```sql
-- 사용자 동적 조건 (user_profiles와 별도 관리)
CREATE TABLE user_conditions (
    user_id         INT REFERENCES user_profiles(id),
    condition_type  VARCHAR(50) NOT NULL,   -- HAS_ARC, HAS_PHONE, HAS_SCHOLARSHIP, etc.
    condition_value BOOLEAN DEFAULT FALSE,
    updated_at      TIMESTAMP DEFAULT NOW(),
    PRIMARY KEY (user_id, condition_type)
);

-- user_profiles에 arc_applied_date 추가
ALTER TABLE user_profiles ADD COLUMN arc_applied_date DATE;

-- university_overrides에 조건부 분기 추가
ALTER TABLE university_overrides ADD COLUMN condition_type VARCHAR(50);
ALTER TABLE university_overrides ADD COLUMN condition_value VARCHAR(100);
-- 예: university_id=연세대, condition_type='HAS_SCHOLARSHIP', condition_value='true'
--     → 장학금 있을 때만 우리은행 계좌 개설 항목 추가

-- checklist_items에 조건부 포함 필드 추가 (동적 항목 추가용)
ALTER TABLE checklist_items ADD COLUMN required_condition VARCHAR(50);
-- NULL = 항상 포함 (기본, 대부분의 항목)
-- 'HAS_SCHOLARSHIP' = 장학금 있을 때만 체크리스트에 포함
-- 'VISA_D2' = D-2 비자일 때만 포함

-- user_checklist_progress에 REMOVED 상태 추가
-- status ENUM: LOCKED, AVAILABLE, IN_PROGRESS, COMPLETED, REMOVED
```

### 실행 단계

```
STEP 1: Load base checklist (조건부 항목 필터링 포함)
─────────────────────────────────────────
SELECT ci.*, os.sort_order as stage_order
FROM checklist_items ci
JOIN onboarding_stages os ON ci.stage_id = os.id
LEFT JOIN user_conditions uc
  ON uc.user_id = :user_id
  AND uc.condition_type = ci.required_condition
WHERE ci.required_condition IS NULL          -- 무조건 포함 (대부분)
   OR uc.condition_value = true              -- 조건 충족 시 포함
ORDER BY os.sort_order, ci.sort_order
→ 조건 매칭 항목만 로드 (기본 15~20개, 조건에 따라 증감)

         │
         ▼

STEP 2: Apply university overrides (조건부 분기 지원)
─────────────────────────────────────────
SELECT override_field, override_value
FROM university_overrides
WHERE university_id = :user_university_id
  AND (condition_type IS NULL                    -- 무조건 적용
       OR (condition_type, condition_value)       -- 조건부 적용
           IN (SELECT condition_type, 
                      CAST(condition_value AS VARCHAR)
               FROM user_conditions
               WHERE user_id = :user_id
                 AND condition_value = true))
→ 학교별 커스텀 적용 (조건 있으면 조건 매칭 후 적용)
예: 서울대 → condition_type=NULL → 무조건 적용 (우리/신한/NH 선택 안내)
    연세대 → condition_type='HAS_SCHOLARSHIP' → 장학금 있을 때만 우리은행 추가

         │
         ▼

STEP 3: Apply bank requirements filter
─────────────────────────────────────────
SELECT * FROM bank_requirements
WHERE visa_type_id = :user_visa_type_id
  AND bank_name = :target_bank
→ 비자별 은행 요건 반영
예: D-2 + 하나은행 → requires_enrollment_cert = true
    → 체크리스트에 "재학증명서 준비" 항목 활성화

         │
         ▼

STEP 4: Evaluate prerequisites (핵심)
─────────────────────────────────────────
FOR EACH checklist_item:
  prereqs = checklist_prerequisites WHERE item_id

  IF already COMPLETED in user_progress:
    status = COMPLETED
  ELSE IF all prereqs satisfied:
    // ITEM type: requires_item_id가 COMPLETED인지
    // CONDITION type: user_conditions에서 값 확인
    status = AVAILABLE
  ELSE:
    status = LOCKED
    blocking_reasons = [미충족 prereq 목록]

         │
         ▼

STEP 5: Sort & return
─────────────────────────────────────────
AVAILABLE items first → LOCKED → COMPLETED
OUTPUT: [{
  id, code, title, description, action_url,
  status, blocking_reasons, estimated_duration
}]
```

### Cascade 예시

```
사용자: "ARC 받았어요"
  │
  ▼
user_conditions["HAS_ARC"] = true
GET_ARC.status = COMPLETED
  │
  ▼
Rule Engine STEP 4 재실행:
  ├─ OPEN_BANK_ACCOUNT: prereq HAS_ARC ✓ + HAS_PHONE ✓ → AVAILABLE ✨
  ├─ GET_POSTPAID_PHONE: prereq HAS_ARC ✓ → AVAILABLE ✨
  ├─ REGISTER_INTERNET_BANKING: prereq HAS_ACCOUNT ✗ → LOCKED (변화 없음)
  └─ 나머지: 변화 없음
```

## 3. AI 비서: Function Calling 설계

### 시스템 프롬프트 (요약)

```
당신은 한국에 온 유학생의 금융 온보딩을 돕는 AI 비서입니다.
체크리스트의 순서와 잠김/해제는 시스템이 관리합니다.
당신의 역할:
1. 사용자의 상태 변화를 감지하면 update_user_status를 호출하세요
2. 질문에 답할 때는 search_knowledge로 검색한 후 답변하세요
3. 체크리스트 관련 질문은 get_checklist, get_item_detail을 사용하세요
4. 금융 조언을 할 때는 반드시 검색 결과에 기반하세요 (환각 금지)
```

### Functions 정의

| Function | 용도 | 호출 조건 |
|----------|------|----------|
| `update_user_status` | 상태 변경 (ARC 수령 등) | "~했어요", "~받았어요" 감지 시 |
| `get_checklist` | 현재 체크리스트 조회 | "지금 뭐 해야 해?" 류 질문 |
| `get_item_detail` | 특정 항목 상세 | "계좌 만들려면?" 류 질문 |
| `search_knowledge` | RAG 검색 | 일반 금융 질문 |

### 대화 흐름 예시

```
사용자: "ARC 받았어요!"
LLM → update_user_status(HAS_ARC=true)
Backend → Rule Engine cascade → 2개 항목 UNLOCK
LLM: "축하해요! 이제 2가지가 가능해졌어요:
      1. ✅ 하나은행 계좌 개설
      2. ✅ 후불 휴대폰 전환
      먼저 어떤 걸 하시겠어요?"

사용자: "계좌 만들려면 뭐 필요해?"
LLM → get_item_detail("OPEN_BANK_ACCOUNT")
    → search_knowledge("하나은행 D-2 계좌 개설")
LLM: "필요 서류: 여권, ARC, 재학증명서
      💡 팁: 이태원 지점이 영어 대응 좋아요
      ⚠️ 주의: 재학증명서 없으면 이체한도 일 30만원
      [하나은행 지점 찾기 →]"
```

## 4. REST API 설계

### 엔드포인트 목록

| Method | Path | 설명 | Auth |
|--------|------|------|------|
| POST | `/api/onboarding` | 온보딩 설문 제출 → 프로필 생성 + 초기 체크리스트 | 없음 (세션 발급) |
| GET | `/api/checklist/:sid` | 현재 체크리스트 상태 조회 | 세션 ID |
| POST | `/api/progress/:sid` | 항목 상태 변경 (start/complete/undo) | 세션 ID |
| POST | `/api/status/:sid` | 사용자 조건 변경 (HAS_ARC 등) | 세션 ID |
| POST | `/api/chat/:sid` | AI 비서 대화 | 세션 ID |
| GET | `/api/item/:sid/:code` | 체크리스트 항목 상세 | 세션 ID |
| GET | `/admin/conflicts` | 파이프라인 상충 정보 대기열 | Admin |
| POST | `/admin/conflicts/:id/resolve` | 상충 해결 | Admin |

## 5. 기술 스택 (MVP)

| 계층 | 기술 | 이유 |
|------|------|------|
| Frontend | Next.js 14 (App Router) | React 기반, SSR, 모바일 반응형 |
| Backend | Next.js API Routes | 프론트와 동일 프로젝트, 배포 단순화 |
| Structured DB | PostgreSQL (Supabase) | 무료 티어, 관계형 쿼리 최적 |
| Vector DB | Chroma (로컬) 또는 Supabase pgvector | MVP는 로컬, 운영은 pgvector |
| LLM | OpenAI GPT-4o-mini | 저렴, function calling 지원 |
| Embedding | OpenAI text-embedding-3-small | 저렴, 한국어 지원 |
| 배포 | Vercel | Next.js 최적화, 무료 티어 |
| 인증 | 없음 (익명 세션 UUID) | MVP 단순화 |

## 6. 데이터 파이프라인 (byul.ai 스타일) 상세

```
[Sources]
    │
    ├── 학교 공지 (Playwright scraper, 주 1회)
    ├── 은행 가이드 (fetch + cheerio, 월 1회)
    ├── Reddit (reddit API / pushshift, 일 1회)
    ├── 정부 사이트 (fetch, 월 1회)
    └── 수동 입력 (admin UI)
    │
    ▼
[Parse]  → LLM으로 구조화 추출 (GPT-4o-mini)
    │      "이 페이지에서 외국인 계좌 개설 조건을 JSON으로 추출하세요"
    │      output: { requires_arc: true, requires_phone: true, ... }
    │
    ▼
[Classify] → 온보딩 단계(S0~S6) + 대상(학교/비자/은행) 자동 태깅
    │
    ▼
[Evaluate] → 신뢰도 점수 부여
    │          공식 사이트(1.0) > 블로그(0.7) > 커뮤니티(0.5)
    │          최신성 가중치: 3개월(1.0), 6개월(0.7), 1년(0.3)
    │
    ▼
[Label] → 메타데이터 태깅 + is_structured 플래그 결정
    │
    ▼
[Dedup] → 동일 정보 병합, 상충 시 conflict_queue에 적재
    │
    ▼
[Split & Store]
    ├── is_structured=true → PostgreSQL (bank_requirements, university_overrides 등 업데이트)
    └── is_structured=false → Embedding → Vector DB (AI Q&A 맥락)

※ 공모전 데모에서는 Phase 1~6을 오프라인 사전 실행 (스크립트로 1회 수행)
   Vector DB는 Reddit 261건 + 3개 대학 공지 정적 스냅샷으로 시드
```

## 7. 동적 항목 추가 메커니즘

체크리스트는 온보딩 시 1회 생성되고 끝나는 게 아니다. 사용자 조건이 바뀌면 항목이 추가되거나 제거된다.

### 트리거 유형

| 트리거 | 예시 | 체크리스트 변화 |
|--------|------|----------------|
| 조건 추가 | 장학금 수령 예정 → HAS_SCHOLARSHIP=true | 장학금 관련 항목 새로 포함 |
| 조건 제거 | 장학금 취소 → HAS_SCHOLARSHIP=false | 해당 항목 REMOVED 처리 |
| 비자 변경 | D-2 → E-7 | 전체 재생성 (기존 COMPLETED 유지, 해당 없는 항목 REMOVED) |
| 정책 변경 | 은행 요건 변경 (파이프라인 감지) | Admin 승인 후 항목 추가/수정 |

### 동작 흐름

```
조건 변경 발생 (채팅 or 프로필 수정)
  │
  ▼
user_conditions 테이블 업데이트
  │
  ▼
Rule Engine 전체 재실행 (STEP 1~5)
  │
  ├─ STEP 1에서 required_condition 매칭으로 항목 셋 변화
  │   • 새로 포함된 항목 → user_checklist_progress에 INSERT (LOCKED or AVAILABLE)
  │   • 더 이상 해당 없는 항목 → status = REMOVED
  │
  ├─ 기존 COMPLETED 항목 → 유지 (변경 없음)
  │
  └─ 응답에 new_items / removed_items 포함
```

### UI 표현

```
새 항목 추가 시:
┌───────────────────────────────────┐
│ 📋 체크리스트                       │
│                                   │
│ ✅ ARC 발급                        │
│ ✅ 하나은행 계좌 개설                │
│ 🔓 인터넷뱅킹 등록                  │
│                                   │
│ ┌─ 🆕 NEW ────────────────────┐  │
│ │ 장학금 수령 계좌 지정          │  │
│ │ 장학금 증빙 서류 준비          │  │
│ └─────────────────────────────┘  │
│                                   │
│ 🔒 체크카드 신청                    │
└───────────────────────────────────┘

항목 제거 시 (비자 변경 등):
┌───────────────────────────────────┐
│ ⚠️ 비자 변경으로 체크리스트가         │
│   업데이트되었습니다                  │
│                                   │
│ ✅ ARC 발급                        │
```

### 비자 변경 시 전체 재생성 로직

```

2. Rule Engine STEP 1~5 전체 재실행
3. 비교 로직:
   기존_항목_셋 = user_checklist_progress의 현재 항목들
   새_항목_셋   = Rule Engine 결과

   교집합 (기존 ∩ 새): 상태 유지 (COMPLETED면 COMPLETED)
   기존에만 있는 것:   status → REMOVED
   새로 추가된 것:     INSERT → LOCKED or AVAILABLE

4. 응답: diff 형태로 변경 내역 전달
```
